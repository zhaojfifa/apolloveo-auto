# 数字人口播产线基础

## 1. 文档定位

本文档用于冻结数字人口播产线在 ApolloVeo 平台中的基础产品定义与平台逻辑，只讨论与平台相关的产品逻辑与定义，不展开工程实现、任务拆分与代码结构细节。其目标是为后续数字人方向的 docs 固化、架构评审、产品讨论与产线规划提供统一口径。

本文档必须服从 ApolloVeo 已有平台基线：平台是一个以成片交付为统一目标、按场景组织独立产线、由公共能力、资产后勤、执行网关与状态治理共同支撑的 AI 内容生产工厂；每条正式产线都应围绕单一主交付结果组织，而不是围绕模型或工具组织。

## 2. 平台定位

ApolloVeo 的平台定位，不是数字人能力集合页，也不是模型调用平台，而是一个以成片交付为统一目标、按场景组织独立产线、由公共能力、资产后勤、执行网关与状态治理共同支撑的 AI 内容生产工厂。平台的组织方式应坚持按产线组织，而不是按模型或工具组织；每条正式产线只面向一个主交付结果，并以标准化交付包和可追溯证据作为协作接口。

在这一平台基线下，数字人方向不应先被定义为大而全数字人平台，而应被定义为平台中的一条标准结果产线，并受同一套平台原则约束：结果优先、交付物优先、资产与产物分区、先打通产线再逐步长成平台。

## 3. 产品线定位

当前阶段，数字人方向的正式产品定义应收敛为：

**面向个人主播的数字人口播视频生产线。**

它不是泛化的视频生成平台，也不是复杂的企业知识视频平台第一阶段版本，而是一条面向个人主播、以脚本与文案为主要输入、以简化角色资产为表达主体、以场景模板装配为核心控制点、以越南语与缅甸语为首批目标语言的可发布成片生产线。

其平台价值不在于生成一个会说话的数字人，而在于把个人主播的表达能力、角色形象、语言适配和场景模板沉淀为可重复调用的生产能力。

## 4. 平台中的归属

数字人口播产品线在平台中的归属，应分为两层理解。

第一，它是一条**结果产线**。
即它的核心目标是稳定交付成片，而不是单独生产角色、单独生产声音、单独生产场景。按照既有平台语义，它应属于结果产线层，和 Hot Follow、Localization、Script Video 等线同属面向 final video 的业务执行单元。

第二，它依赖一组**公共供给能力**。
角色、声音、语言预设、场景模板、参考风格、执行路由等，不应被等同为结果产线本身，而应被理解为供给支撑层和资产层向这条产线提供的可复用能力。这样，数字人产品线才能保持主结果单一、业务目标清晰，而不会把资产、能力、场景和结果揉成一个不稳定的大对象。

## 5. 面向平台的产品逻辑

这条产品线的核心产品逻辑，不是先做角色，再拼视频，而是：

**将输入的脚本/文案转化为可执行的表达结构，再通过场景模板和数字人绑定机制，稳定生产出可发布成片。**

因此，它的业务核心不在角色资产有多复杂，而在三件事。

### 5.1 内容先被结构化，而不是直接被渲染

平台层面，脚本和文案不是终态输入，而是生产原料。系统需要先把它们转化为可执行的表达结构，例如讲解提纲、段落逻辑、节奏安排、强调点和输出语言要求。

### 5.2 场景模板是核心控制点，而不是附属素材

对个人主播场景而言，真正决定平台可复用性和可规模化的，不只是角色本身，而是场景模板、镜头布局、背景样式、讲解区与信息区的结构，以及是否支持参考生成。也就是说，这条产线在平台上最重要的可复用对象之一，不只是角色资产，而是模板化的 scene plan 与 scene template。

### 5.3 数字人是表达载体，不是产品本体

在平台定义中，数字人不是孤立存在的主产品，而是承担表达、讲解、呈现的统一角色。真正的平台产品对象，是内容结构、场景模板、角色绑定、语言输出与交付包的组合体。这样定义后，平台才能自然支持后续的多语言、本地化、版本化与 profile 扩展。

## 6. 当前阶段的范围收口

平台层面，当前这条数字人口播产品线应主动收窄，而不是追求行业完整版。

### 6.1 目标用户收窄

第一阶段面向个人主播，而不是同时覆盖企业培训、机构知识视频、品牌全量内容中台。这样可以把产品逻辑集中在个人主播如何稳定生成可发布口播视频，而不是一开始进入企业级复杂输入和复杂治理。

### 6.2 语言范围收窄

第一阶段只聚焦越南语与缅甸语。

这不是简单的语言裁剪，而是整条产品线在 voice preset、字幕规范、语言预设、输出质量控制上的统一收口。语言越窄，平台越容易形成稳定的标准线。

### 6.3 输入范围收窄

第一阶段的主输入，不追求 document、PPT、URL、knowledge base 的全入口，而是以脚本、文案和简短场景说明为主，并允许可选的参考图或参考视频。这样既保留了输入先结构化的经典逻辑，又避免在平台基座尚未完全收口时把输入层做得过宽。

### 6.4 角色资产收窄

第一阶段角色资产不追求复杂角色工厂，而以个人主播自身形象、声音、身份设定的简化 profile 为主。角色资产在平台中的职责，是支撑表达一致性，而不是先扩展成复杂的数字人资产市场。

## 7. 平台侧的核心对象定义

为了让这条产线在平台中具备清晰边界，建议冻结以下产品对象定义。

### 7.1 Role Profile

个人主播的简化角色对象，包含形象、声音、身份设定与基础表达风格。它是角色绑定对象，不是成片对象。

### 7.2 Presentation Outline

内容结构对象。用于把脚本或文案整理成可执行的讲解结构，包含主题、段落、要点、节奏和强调信息。它是内容层真相，不是前端临时状态。

### 7.3 Scene Plan

场景结构对象。用于定义背景、模板、布局、信息区、参考风格与画面分段关系。它是场景层真相，决定画面如何稳定生成。

### 7.4 Speaker Plan

表达绑定对象。用于把角色、声音、节奏和口播方式绑定到内容结构上。它是“谁来讲、怎么讲”的定义对象。

### 7.5 Language Plan

语言输出对象。用于定义目标语言、字幕策略、语音语言、术语规则与输出版本要求。当前阶段以越南语与缅甸语为主。

### 7.6 Delivery Pack

统一交付对象。平台不应把产物理解为单一 mp4，而应理解为 final video 加 subtitle、audio、metadata、manifest、pack 等构成的交付包。

## 8. 统一输入与输出口径

### 8.1 输入口径

这条产品线的第一阶段输入应统一收敛为四类：

- 脚本 / 文案
- 角色 profile
- 场景模板 / 场景说明
- 目标语言

可选附加输入为参考图或参考视频，但不应成为第一阶段的主入口。

### 8.2 输出口径

这条产品线的输出不应只定义为一个数字人视频，而应定义为：

- 主交付物：final video
- 附属交付物：subtitle、audio、metadata
- 平台证据物：manifest、pack、lineage
- 可复用上下文：role usage、scene usage、language usage

## 9. 生产逻辑定义

平台层面，这条产线的生产逻辑应冻结为：

**输入归一化 → 内容结构生成 → 场景/模板装配 → 数字人绑定 → 语音/口型执行 → 成片合成 → 本地化/版本化输出**

这里要特别强调：这条链是**生产逻辑**，不是前端配置顺序。前端可以有分步配置，但平台定义必须围绕“内容如何被结构化并最终变成可交付成片”来组织，而不是围绕页面表单顺序组织。这样才能保证后续 Workbench、Scenario Spec 和交付逻辑一致。

## 10. 与其他产线的关系

### 10.1 与 Script Video 的关系

数字人口播产品线，本质上是 Script Video 在个人主播口播场景下的收窄和平台化表达。它不替代 Script Video 的上位定义，而是当前更适合先落地的窄场景标准线。

### 10.2 与 Localization 的关系

多语言输出是这条产线的重要能力，但这条线本身不等于 Localization。Localization 仍是独立产线；数字人口播产品线只是当前阶段把越南语与缅甸语作为首批目标语言纳入自身标准输出范围。

### 10.3 与 Avatar Commerce Profile 的关系

Avatar Commerce 不应先作为独立正式产线存在，而应作为这条数字人口播产品线之上的业务 profile。也就是说，平台先冻结个人主播数字人口播成片这条主线，再在其上扩展带货表达、商品信息、参考生成和版本化放大。这样更符合先打通产线，再抽象平台的平台原则。

## 11. 一句话定义

**数字人口播产品线，是平台中的一条标准结果产线：它面向个人主播，以脚本与文案为主要输入，以简化角色资产为表达主体，以场景模板为核心控制点，以越南语与缅甸语为首批目标语言，通过“输入归一化 → 内容结构生成 → 场景装配 → 数字人绑定 → 语音/口型执行 → 成片合成 → 本地化/版本化输出”的生产逻辑，交付可发布成片及其标准化交付包。**
