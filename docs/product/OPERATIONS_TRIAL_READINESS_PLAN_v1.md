## 0.1 Matrix Script Trial Sample Validity (binding, post-§8.A / §8.B / §8.C / §8.D / §8.E / §8.F / §8.G / §8.H)

This section is the operator-facing rule on which Matrix Script samples count as trial evidence. It supersedes the prior §0.1 wording that was authored only against §8.A / §8.B / §8.C; §3 / §6 / §7 sub-sections below have been re-corrected to match the full chain. The pre-§8.F live-trigger sample (`task_id=415ea379ebc6`, which submitted a publisher article URL `https://news.qq.com/rain/a/...`) is also invalid evidence per the follow-up blocker review §1 / §9.

- **Old broken Matrix Script sample (pre-§8.A): INVALID.** Any Matrix Script task created before the §8.A `source_script_ref` ref-shape guard landed is **invalid trial evidence** and MUST NOT be reused. The blocker review §9 ruling stands; §8.A's PASS gate explicitly does not retrofit existing rows. If an operator session encounters a task whose `source_url` carries pasted script body text (multi-line, prose, or any non-opaque-ref payload), it is the prior invalid sample — discard it from trial evidence and create a fresh sample via the formal route.
- **Pre-§8.F live-trigger sample (publisher URL): also INVALID.** Any Matrix Script task whose `source_url` / `packet.config.entry.source_script_ref` carries an `https://` / `http://` URL, an `s3://` / `gs://` bucket URI, or any other non-opaque-by-construction scheme is **invalid trial evidence** under the §8.F-tightened scheme set. The follow-up review §1 / §3 documents the live-trigger sample (`task_id=415ea379ebc6`) — the §8.F guard would now reject the same input at HTTP 400. Discard from trial evidence and create a fresh sample via the formal route under the tightened scheme set (see §0.2 below).
- **Fresh corrected Matrix Script sample (post-§8.A / §8.B / §8.C / §8.D / §8.E / §8.F / §8.G): the only acceptable trial sample type.** A sample qualifies as trial evidence iff **all seven** of the following hold:
  1. **Formal create-entry route.** It was created via the formal `/tasks/matrix-script/new` POST (the formal Matrix Script entry path) — never the generic `/tasks/connect/...` path.
  2. **§8.A + §8.F opaque-ref shape.** Its persisted `source_url` (and `packet.config.entry.source_script_ref`) is an opaque ref handle accepted by the §8.A guard with the §8.F-tightened scheme set: a URI whose scheme is one of the four opaque-by-construction schemes `content` / `task` / `asset` / `ref`, or a bare token id matching the documented pattern (`^[A-Za-z0-9][A-Za-z0-9._\-:/]{3,}$`); single line; no whitespace; ≤512 chars. **External web schemes (`https`, `http`) and bucket schemes (`s3`, `gs`) are no longer accepted** — they were dropped by §8.F because they are not opaque-by-construction inside the product.
  3. **§8.F operator transitional convention recommended.** During the Plan A trial wave the canonical operator-facing form is `content://matrix-script/source/<token>` where `<token>` is operator-chosen and uniquely identifies the operator's script source within the operator's own bookkeeping. The product treats `<token>` as fully opaque (never dereferences, never validates uniqueness, never reads back content); operators maintain their own private mapping between `<token>` and the source location (working notes, internal page, private doc, etc.). Other `task://` / `asset://` / `ref://` opaque shapes are also accepted; the `content://matrix-script/source/<token>` form is the recommended convention for new trial samples in this wave.
  4. **§8.C populated Phase B truth.** Its packet carries populated `line_specific_refs[matrix_script_variation_matrix].delta` (3 canonical axes — `tone` / `audience` / `length` — and `variation_target_count` cells) and `line_specific_refs[matrix_script_slot_pack].delta` (one slot per cell) authored deterministically at task creation per §8.C; `cells[i].script_slot_ref ↔ slots[i].slot_id` round-trips for every i; `body_ref = "content://matrix-script/{task_id}/slot/{slot_id}"` is opaque by construction.

## 0.2 Product Meaning of `source_script_ref` (binding, §8.H)

This section is the **product-meaning** statement that operators MUST internalize before submitting any Matrix Script task. It is required by §8.H of the follow-up blocker review v1 and is the conceptual foundation for the §8.A / §8.F shape rules in §0.1.

- **What `source_script_ref` IS.** A **source-script asset identity handle**. It identifies *which source-script asset* this Matrix Script task is about — analogous to a product SKU, a row id, or any other opaque identity label. The operator picks (or mints, via their own bookkeeping) an opaque token uniquely identifying the script source in their own working notes, then submits that token wrapped in a `content://matrix-script/source/...` URI (or one of the other three opaque schemes `task://` / `asset://` / `ref://`). The product stores this handle on the persisted task as the operator's chosen identity; downstream Phase B / Phase C / Phase D flows treat it as opaque.
- **What `source_script_ref` IS NOT.**
  1. **NOT a body-input field.** It is not where script body text, prose, document content, scene descriptions, or any narrative text is supposed to be entered. The §8.A guard rejects any value that contains whitespace, multi-line content, or exceeds 512 chars precisely because none of those shapes are identity handles. If you have a script body, it lives in your own working notes — not in this field.
  2. **NOT a publisher-URL ingestion field.** It is not a "fetch this article" or "scrape this URL" instruction. It is not the URL of the news article you're adapting from; it is not the URL of the YouTube video you're matrix-scripting around; it is not the link to the Notion / Lark / Google Doc that hosts your source. The §8.F tightening explicitly removed `https://` / `http://` from the accepted scheme set because the product does not (and will not in this wave) dereference external web URLs. Submitting a publisher URL is now rejected at HTTP 400 with `"scheme is not recognised"`.
  3. **NOT a currently dereferenced content address.** Even with the four retained opaque-by-construction schemes (`content://` / `task://` / `asset://` / `ref://`), the product does not currently dereference the handle to body content during the Plan A trial wave. The handle is stored verbatim. The §8.C deterministic Phase B authoring step uses the handle only as part of the per-task identity envelope; it does not read script content out of the handle. The Plan B B4 (`result_packet_binding_artifact_lookup_contract_v1`) artifact-lookup mechanism remains contract-only (gated to Plan E); no consumer in this wave reads body content via `source_script_ref` resolution. An operator who expects "the system fetched my source script body via this handle" would be wrong about the current product.
- **Why this matters for the trial.** The combination of "asset identity handle, not body input, not URL ingestion, not currently dereferenced" defines exactly what operators may safely submit. Any deviation produces invalid trial evidence: pasting body fails §8.A; pasting a URL fails §8.F; expecting downstream to resolve content fails §0.2 point 3. Coordinator MUST brief operators on this distinction verbatim before any sample-creation session.
- **Forward compatibility with Plan E (Option F2).** When Plan E adds the in-product minting flow (Option F2 — see follow-up review §6 and the §8.F contract addendum's `Operator transitional convention` sub-section), the operator-discipline-only `<token>` convention is replaced by a product-minted handle and the product begins to own the dereferencing path. The shape of the field — opaque ref handle — does not change. Any handle minted under the §8.H convention remains valid forward (it is one instance of the `content://` opaque scheme); operators are not asked to migrate.
